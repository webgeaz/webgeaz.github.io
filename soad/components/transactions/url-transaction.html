    URL Transactions

    This document provides an overview of URL structures and directory details for managing transactions in the system.

    Overview

    Before starting application development, it is essential to establish a Group. A Group acts as a module, organizing related Transactions, Data, and Queries in one place. This structured approach enhances maintainability, improves readability, and ensures better scalability.

    Each Group is typically defined based on user requirements or business needs. Every Transaction, Data, and Query within a Group must have a corresponding Code, forming a unique ID and URL. The system follows this structure to ensure consistency and easy access to different transactions.

    By maintaining a clear directory structure and standardized URL format, the system provides an efficient way to manage transactions and their associated views.

    Directory Structure

    Each transaction is stored in a specific directory following a structured hierarchy:

    1. Parent Directory (Group Folder)

      • Represents a group of transactions.
      • Contains multiple transaction files with the .py extension.

    2. Transaction File

      • Each transaction is stored as a .py file within its respective group.

    3. Transaction Subdirectory

      • A subdirectory is created with the same name as the transaction file.
      • This subdirectory contains all the related views for that transaction.

    4. Default View File

      • The default view for a transaction is an .html file.
      • The .html file is named identically to the transaction file to maintain consistency.

    URL Structure

    The system follows a standardized URL structure to access transactions and their views efficiently.

    Base URL Structure

    /t/{group_name}/{code_name}/{action_name}

    or follow the image above

    URL Components

    1. /t (Transaction Base)

      • The base URL prefix for all transactions.
      • This is mandatory and must be included before any transaction URL.

    2. /{group_name} (Group Name)

      • Represents the parent directory (group) where the transaction is stored.
      • Helps in organizing transactions into different categories.

    3. /{code_name} (Code Name)

      • Represents the specific code that has been created.
      • Matches the name of the .py transaction file.

    4. /{action_name} (Action Name)

      • If there are no action name in the transaction or URL, /view will be returned as default.
      • Appending /edit or any other action name in the URL refers to a specific function inside the code file.

    Example URLs

    Accessing a Code

    In the code of frontend, you can access the transaction by using the following URL:

    {{ctxPath}}/t/{group_name}/{code_name}
    but starting with this URL and then append the group name, code name, and action name.

    if you want to access the view of the transaction, you can directly access it by using the following URL:

    https://soad.dev1.web/t/{group_name}/{code_name}/view

    1. /t/base/newtrans

      • t/ → Base URL for transactions.
      • base/ → Group (parent directory).
      • newtrans/ → Transaction name.

    2. /t/base/newtrans/view

      • /view → Calls the function responsible for rendering the view.

    3. /t/base/newtrans/edit?name="Yaakub" with one parameter

      • /edit → Calls the function responsible for editing the transaction.
      • ?name="Yaakub" → Additional parameters passed to the function with one parameter.

    4. /t/base/newtrans/edit?name="Yaakub"&age=25 with multiple parameters

      • & → act as separator for multiple parameters
      • age=25 → Additional parameters passed to the function with multiple parameters.

    Getting Data from the Request and Response

    This section to describes the how to manage and get the data from the request and response. The ctx object is used to process the request and response.

    1. Get the Request Data

    The ctx object is used to get the data from the request.

    request = ctx.getRequest()                  # Get the request object
user_id = request.getParameter("user_id")   # Get the user_id from the request

    2. Get the Data from Database

    Getting data from database in SØAD is used activeJDBC.

    from models import Tb_user 

# Rest of the code
tb_user = Tb_user.findById(user_id) 
name = tb_user.get("name")          # Get the name from the user object

    Tip

    In import the model class need to be Uppercase on the first letter for example Tb_user.

    3. Set the Response Data to The Database

    The ctx object is used to set the data to the database. As example, the table name tb_user:

    tb_user.set("name", "SØAD")     # Set the name to the user object
tb_user.saveIt()                # Save the user object to the database

    Tip

    There are two ways to save the data to the database:

    • saveIt(): Automatically saves the model.

      • If the model is new, it runs an INSERT SQL.
      • If the model already exists, it runs an UPDATE.
      • Best used when your table uses AUTO_INCREMENT for the primary key.

    • insert(): Inserts new data into the database.

      • Only performs an INSERT operation.
      • Useful when the primary key is manually set (either by the system or user).

    Display the Output in View

    The ctx object can be used to display the output in the view. The Handlebars will render the output from the ctx.

    Usage Examples

    backend.py
    ctx.output["name"] = "SØAD"
    index.html
    <div>
    <h2>
        {{name}}
    </h2>
</div>
